# 데이터 처리와 분석 로직

이 문서는 **When Whales Move** 프로젝트에서 데이터를 어떻게 가져오고, 정제하고, 분석했는지를 기술적으로 설명한다. 발표나 코드 설명에서 “데이터 분석 로직이 어디에 있고 어떤 방식으로 계산됐는가”를 설명할 때 사용할 수 있다.

## 1. 전체 처리 흐름

이 프로젝트의 데이터 흐름은 다음 순서로 구성된다.

```text
원천 CSV / 크롤링 데이터
        ↓
Python 파이프라인에서 로드
        ↓
정제 및 표준화
        ↓
이벤트 기준 ±7일 윈도우 생성
        ↓
PostgreSQL 적재
        ↓
Fastify API 조회
        ↓
React 대시보드에서 가설 검증 및 시각화
```

핵심은 분석 계산을 화면에서 즉석으로 전부 처리하지 않고, **파이프라인에서 분석용 테이블을 먼저 만들고**, 프론트엔드는 그 결과를 읽어서 해석과 시각화를 담당하도록 나눈 것이다.

## 2. 사용한 데이터

| 데이터 | 주요 파일 / 테이블 | 사용 목적 |
|---|---|---|
| 고래 송금 데이터 | `whale_data/whales_*.csv`, `whale_transfers` | 큰 지갑의 대규모 자금 이동 추적 |

| ETH/BTC 가격 데이터 | `price_data/eth_price.csv`, `price_data/btc_price.csv`, `prices` | 사건 전후 가격 반응 비교 |
| 이벤트 메타데이터 | `db/seeds/events.sql`, `events` | 분석 기준이 되는 사건 날짜와 분류 |
| 뉴스 데이터 | `news_articles` | 사건 맥락과 뉴스 볼륨 제공 |

| 테이블 | 행 수 |
|---|---:|
| `whale_transfers` | 1,286 |
| `cex_flows` | 15,393 |

## 3. 파이프라인 구조

데이터 처리의 중심은 `pipelines/pipelines/` 디렉토리다.

```text
pipelines/pipelines/
├── ingest/       # CSV 로드
├── transform/    # 정제, 정렬, 이벤트 윈도우 생성
├── load/         # PostgreSQL 적재
├── run_all.py    # 전체 파이프라인 실행
├── config.py     # 입력 파일 경로와 DB URL 설정
└── db.py         # SQLAlchemy DB 엔진 생성
```

전체 파이프라인은 `pipelines/pipelines/run_all.py`에서 다음 순서로 실행된다.

1. 고래 송금 CSV 로드
2. 거래소 입출금 CSV 로드
3. 가격 CSV 로드
4. 각 데이터 검증
5. 원천 테이블 적재
6. 이벤트 목록 조회
7. 이벤트 윈도우 생성
8. 이벤트 윈도우 검증 및 적재
9. DB에 저장된 값과 파이프라인 결과가 일치하는지 재검증
10. 테이블별 행 수 스냅샷 저장

## 4. 데이터 로드 방식

### 4.1 고래 송금 데이터

파일:
- `pipelines/pipelines/ingest/whale_csv_loader.py`

고래 송금 데이터는 `whale_data/whales_*.csv` 형식의 CSV를 읽는다.

주요 컬럼은 다음과 같이 매핑된다.

| 원본 컬럼 | 내부 의미 |
|---|---|
| `tx_week` | 주 시작일 |
| `asset` | 자산 종류 |
| `whale_tx_count` | 고래 송금 횟수 |
| `total_amount` | 총 전송량 |
| `total_usd_volume` | 달러 기준 총 전송액 |

이 데이터는 주간 단위이기 때문에 이후 이벤트 윈도우를 만들 때 일 단위로 나눠서 사용한다.

### 4.2 거래소 입출금 데이터

파일:
- `pipelines/pipelines/ingest/cex_flow_loader.py`

거래소 입출금 데이터는 Dune에서 추출한 CSV를 읽는다.

주요 컬럼은 다음과 같다.

| 컬럼 | 의미 |
|---|---|
| `flow_date` | 입출금 발생일 |
| `asset` | 자산 종류 |
| `direction` | `inflow` 또는 `outflow` |
| `transfer_count` | 전송 횟수 |
| `total_volume_native` | 자산 단위 총량 |
| `total_volume_usd` | 달러 기준 총액 |

분석에서는 `direction`을 기준으로 다음처럼 해석한다.

- `inflow`: 외부 지갑에서 거래소로 들어간 자금
- `outflow`: 거래소에서 외부 지갑으로 빠져나간 자금

### 4.3 가격 데이터

파일:
- `pipelines/pipelines/ingest/price_loader.py`

가격 데이터는 ETH와 BTC 가격 CSV를 함께 읽는다.

| 컬럼 | 의미 |
|---|---|
| `date` | 날짜 |
| `price` | 달러 기준 가격 |
| `volume` | 거래량 |
| `market_cap` | 시가총액, 있으면 사용 |

현재 대시보드의 핵심 시각화는 ETH 가격을 중심으로 사용하고, BTC 가격은 DB와 API에는 포함되어 향후 비교 분석에 확장할 수 있게 남겨둔 상태다.

## 5. 정제 및 검증

정제 로직은 `pipelines/pipelines/transform/`에 있다.

### 5.1 고래 송금 정규화

파일:
- `pipelines/pipelines/transform/whale_transfers.py`

처리 내용:

1. 2017년 이전 데이터 제외
2. `(week_start, asset)` 기준으로 중복 행 병합
3. 송금 횟수와 총 전송액 합산
4. 날짜와 자산 기준으로 정렬

이 처리를 하는 이유는 주간 경계나 원천 CSV 분리 과정에서 같은 주·같은 자산 조합이 중복될 수 있기 때문이다.

### 5.2 거래소 입출금 정렬

파일:
- `pipelines/pipelines/transform/cex_direction.py`

거래소 입출금 데이터는 날짜, 자산, 방향 기준으로 정렬한다.

```text
(flow_date, asset, direction)
```

정렬 자체는 단순하지만, 이후 DB 적재와 검증에서 항상 같은 순서를 보장하는 역할을 한다.

### 5.3 데이터 유효성 검증

파일:
- `pipelines/pipelines/transform/validators.py`

검증 조건:

| 데이터 | 검증 조건 |
|---|---|
| 고래 송금 | 금액이 음수가 아니어야 함, 2017년 이후여야 함 |
| 거래소 입출금 | 방향은 `inflow` 또는 `outflow`, 금액은 음수가 아니어야 함 |
| 가격 | 가격은 0보다 커야 함 |
| 이벤트 윈도우 | `day_offset`은 -7부터 +7 사이, 공포·탐욕 지수는 0~100 |

즉, 분석 전에 비정상 값이 들어오면 바로 실패하도록 했다.

## 6. 이벤트 윈도우 생성

파일:
- `pipelines/pipelines/transform/event_windows.py`

이벤트 윈도우는 **사건일을 기준으로 전후 7일 데이터를 잘라낸 분석 단위**다.

```text
D-7 D-6 ... D-1 | D0 | D+1 ... D+7
```

각 이벤트마다 총 15일 구간을 만들고, 각 날짜에 다음 지표를 붙인다.

| 지표 | 설명 |
|---|---|
| `whale_volume_usd` | 해당 날짜의 고래 송금액 |
| `cex_inflow_usd` | 해당 날짜의 거래소 입금액 |
| `cex_outflow_usd` | 해당 날짜의 거래소 출금액 |
| `eth_price_usd` | 해당 날짜의 ETH 가격 |
| `btc_price_usd` | 해당 날짜의 BTC 가격 |
| `fear_greed_value` | 해당 날짜의 공포·탐욕 지수 |

### 6.1 고래 주간 데이터의 일 단위 변환

고래 송금 데이터는 주간 단위이므로 이벤트 윈도우에 붙이기 위해 7일로 균등 분배한다.

```text
하루 고래 송금액 = 주간 고래 송금액 / 7
```

예를 들어 어떤 주의 고래 송금액이 70억 달러라면, 해당 주의 각 날짜에 10억 달러씩 배분한다.

이 방식은 일자별 정확한 송금 시각을 복원하는 것이 아니라, 주간 단위 데이터를 이벤트 전후 일자 흐름에 맞춰 비교하기 위한 근사 처리다.

### 6.2 거래소 입출금 결합

거래소 입출금 데이터는 이미 일 단위이므로, 날짜와 방향으로 바로 결합한다.

```text
(window_date, inflow)  → cex_inflow_usd
(window_date, outflow) → cex_outflow_usd
```

### 6.3 가격 데이터 결합

ETH와 BTC 가격은 날짜와 자산 기준으로 결합한다.

```text
(window_date, ETH) → eth_price_usd
(window_date, BTC) → btc_price_usd
```

## 7. PostgreSQL 적재 방식

적재 로직은 `pipelines/pipelines/load/`에 있다.

주요 특징은 **업서트** 방식이다.

```sql
INSERT ...
ON CONFLICT (...) DO UPDATE SET ...
```

즉, 같은 파이프라인을 다시 실행해도 중복으로 행이 쌓이지 않고 기존 값이 갱신된다. 이 때문에 파이프라인을 반복 실행해도 같은 결과를 유지할 수 있다.

| 적재 파일 | 대상 테이블 | 충돌 기준 |
|---|---|---|
| `load/whale_transfers.py` | `whale_transfers` | `(week_start, asset)` |
| `load/cex_flows.py` | `cex_flows` | `(flow_date, asset, direction)` |
| `load/prices.py` | `prices` | `(price_date, asset)` |
| `load/event_windows.py` | `event_windows` | `(event_id, day_offset)` |

DB 연결과 SQL 실행은 SQLAlchemy를 사용한다. 다만 ORM 모델을 복잡하게 쓰는 방식이 아니라, DB 엔진과 트랜잭션을 열고 raw SQL을 안전하게 실행하는 용도에 가깝다.

## 8. DB 검증 방식

파일:
- `pipelines/pipelines/run_all.py`
- `pipelines/pipelines/load/reconcile.py`

파이프라인은 데이터를 적재한 뒤, DB에 실제 저장된 값이 파이프라인에서 계산한 값과 일치하는지 확인한다.

검증 방식:

1. 파이프라인에서 기대 행 목록 생성
2. DB에서 실제 행 조회
3. Decimal 값을 DB 스케일에 맞게 반올림
4. 기대 행과 실제 행을 정렬 후 비교
5. DB에만 남아 있는 오래된 행은 `prune_stale_rows`로 제거

이렇게 해서 CSV나 계산 로직이 바뀌었을 때도 DB가 이전 결과를 계속 들고 있지 않도록 했다.

## 9. 뉴스와 공포·탐욕 지수 처리

뉴스와 심리 지표는 자금 흐름을 해석하기 위한 보조 맥락으로 사용한다.

### 9.1 뉴스 수집

관련 파일:
- `pipelines/crawl/core.py`
- `pipelines/crawl/jobs/submit_all.py`
- `pipelines/crawl/jobs/collect.py`
- `pipelines/crawl/parsers/`
- `pipelines/crawl/utils.py`

뉴스는 Cloudflare Browser Rendering `/crawl` API를 통해 수집한다.

처리 흐름:

1. 소스 설정 파일에서 크롤 대상 URL 로드
2. Cloudflare `/crawl`에 job 제출
3. job 결과를 Markdown으로 저장
4. 각 언론사별 parser로 기사 후보 추출
5. 기사 URL, 제목, 날짜, 언어, 요약을 정규화
6. 이벤트와 매칭 후 `news_articles` 테이블에 적재

### 9.2 뉴스와 이벤트 매칭

파일:
- `pipelines/crawl/utils.py`

기사와 이벤트를 매칭할 때는 다음 기준을 사용한다.

1. 이벤트의 `source_url`과 기사 URL이 같으면 바로 매칭
2. 기사 URL이나 메타데이터에서 날짜를 추출
3. 기사 날짜가 이벤트 날짜 기준 ±7일 안에 있는지 확인
4. 이벤트 이름/설명과 기사 제목/URL에서 키워드를 추출
5. 키워드 겹침이 있는 후보 중 가장 가까운 이벤트 선택

즉, 단순히 날짜만 보는 것이 아니라 **날짜 + 키워드 겹침**을 함께 사용한다.

### 9.3 공포·탐욕 지수

관련 파일:
- `pipelines/crawl/load.py`

공포·탐욕 지수는 공개 API에서 가져와 `fear_greed_index` 테이블에 저장한다.

이후 `backfill_event_windows()`에서 `event_windows`의 각 날짜에 맞는 심리 지수를 채운다.

## 10. API에서 제공하는 분석 데이터

API는 PostgreSQL을 읽어서 프론트에 전달한다.

주요 파일:
- `apps/api/src/repos/events.ts`
- `apps/api/src/repos/series.ts`

주요 엔드포인트:

| 엔드포인트 | 설명 |
|---|---|

4. 사건 전 대비 변화율 계산
5. 정규화 지수 계산
6. 거래소 순유입 계산
7. 가설 카드 판정
8. 카테고리/지역별 비교 데이터 생성

## 12. 변화율 계산

각 지표는 사건 전 7일 평균을 기준선으로 사용한다.

```text
변화율(%) = (현재값 - 사건 전 7일 평균) / 사건 전 7일 평균 × 100
```

예를 들어 사건 전 평균 거래소 입금액이 10억 달러이고, 사건일 입금액이 15억 달러라면:

```text
(15억 - 10억) / 10억 × 100 = +50%
```

이 값은 “사건일 거래소 입금이 평소보다 50% 강했다”는 의미로 해석한다.

## 13. 정규화 지수 계산

상세 차트에서는 서로 단위가 다른 지표를 비교하기 위해 사건 전 7일 평균을 100으로 둔다.

```text
정규화 지수 = 현재값 / 사건 전 7일 평균 × 100
```

해석:

| 값 | 의미 |
|---:|---|
| 100 | 사건 전 7일 평균과 동일 |
| 150 | 평소보다 50% 큼 |
| 80 | 평소보다 20% 작음 |

이 방식을 사용한 이유는 고래 송금액, 거래소 입금, 거래소 출금의 절대 규모가 서로 다르기 때문이다. 모두 같은 기준선 100으로 바꾸면 “어떤 지표가 평소 대비 더 크게 반응했는가”를 쉽게 비교할 수 있다.

---

## 14. 거래소 순유입 계산

거래소 순유입은 다음과 같이 계산한다.

```text
거래소 순유입 = 거래소 입금액 - 거래소 출금액
```

해석:

| 값 | 의미 |
|---:|---|
| 양수 | 거래소로 들어간 자금이 더 큼 |
| 음수 | 거래소 밖으로 빠져나간 자금이 더 큼 |
| 0 근처 | 입금과 출금이 비슷함 |

이 지표를 넣은 이유는 입금과 출금을 따로 보면 흐름이 복잡할 수 있기 때문이다. 순유입을 보면 사건 전후 자금 방향이 거래소 안쪽인지 바깥쪽인지 더 직관적으로 볼 수 있다.

---

## 15. 가설 검증 로직

가설 카드는 `apps/web/src/lib/eventAnalysis.ts`에서 자동 생성된다.

대표 가설은 다음과 같다.

| 가설 | 검증 기준 |
|---|---|
| 위기 이벤트에서는 사건일 거래소 입금이 증가한다 | 사건일 입금액이 사건 전 7일 평균보다 얼마나 증가했는지 |
| 호재 이벤트에서는 출금 반응이 입금보다 강하다 | 사건일 출금 변화율과 입금 변화율 비교 |
| 큰 위기에서는 입금과 출금이 함께 증가하며 자금 재배치가 나타난다 | 입금 변화율과 출금 변화율이 둘 다 증가하는지 |
| 사건 뒤 가격 흐름이 시장 해석을 뒷받침한다 | 사건 후 7일 평균 ETH 가격이 기대 방향으로 움직였는지 |

판정은 `supported`, `mixed`, `inconclusive`, `rejected` 네 단계로 나눈다.

증가 방향 가설의 기준:

| 변화율 | 판정 |
|---:|---|
| `+20%` 이상 | 지지 |
| `+5%` 이상 | 부분 지지 |
| `-5%` 초과 | 판정 보류 |
| 그 이하 | 반대 |

하락 방향 가설의 기준:

| 변화율 | 판정 |
|---:|---|
| `-10%` 이하 | 지지 |
| `-3%` 이하 | 부분 지지 |
| `+3%` 미만 | 판정 보류 |
| 그 이상 | 반대 |

이 기준은 통계적 인과관계를 증명한다기보다, 대시보드에서 사건별 반응 강도를 일관되게 설명하기 위한 규칙 기반 판정이다.

## 16. 카테고리/지역 비교 분석

비교 탭에서는 개별 이벤트가 아니라 이벤트 그룹의 평균 반응을 본다.

파일:
- `apps/web/src/lib/eventAnalysis.ts`
- `apps/web/src/features/compare/CategoryCompareView.tsx`

처리 방식:

1. 각 이벤트의 이벤트 윈도우를 가져온다.
2. 이벤트별로 고래 송금, 입금, 출금, 가격 변화율을 계산한다.
3. 이벤트 카테고리 또는 지역별로 묶는다.
4. 같은 그룹 안의 변화율 평균을 낸다.
5. 히트맵으로 표시한다.

카테고리 기준:

- 폭락
- 위기
- 호재
- 과열
- 규제

지역 기준:

- 글로벌
- 한국

히트맵을 사용한 이유는 정확한 숫자 하나보다 “어떤 유형에서 어떤 지표가 강하게 반응했는가”라는 패턴을 빠르게 읽기 위해서다.

---

## 17. 한국 이벤트 비교 분석

파일:
- `apps/web/src/features/korea/KoreaSectorView.tsx`

한국 섹터 화면에서는 한국 이벤트만 따로 골라 small multiples 형태로 비교한다.

분석 방식:

1. `region = kr`인 이벤트만 선택
2. 각 이벤트의 ±7일 윈도우 조회
3. 거래소 입금/출금 값을 사건 전 7일 평균 = 100으로 정규화
4. 같은 축 기준으로 작은 차트를 나란히 표시
5. 데이터가 없는 이벤트는 빈 차트처럼 보이지 않게 명시적으로 “데이터 없음” 처리

이 화면의 목적은 “국내 이벤트가 글로벌 이벤트만큼 강하게 반응했는가”를 비교하는 것이다.

## 18. 분석 결과를 시각화하는 방식

| 화면 | 사용 데이터 | 분석 목적 |
|---|---|---|
| 메인 타임라인 | 고래 송금 + 이벤트 날짜 | 전체 기간에서 큰 사건과 자금 흐름을 같이 보기 |

| 거래소 순유입 그래프 | 입금 - 출금 | 자금 방향이 거래소 안쪽인지 바깥쪽인지 보기 |
| 시장 컨텍스트 차트 | ETH 가격, 심리, 뉴스 | 자금 흐름을 해석할 배경 보기 |
| 비교 히트맵 | 그룹별 평균 변화율 | 이벤트 유형별 반응 패턴 비교 |
| 한국 섹터 | 한국 이벤트 윈도우 | 국내 이벤트끼리 반응 강도 비교 |

## 19. 기술적 한계와 해석 주의점

### 19.1 인과관계가 아니라 사건 전후 반응 분석

이 프로젝트는 “이 사건 때문에 반드시 이런 자금 흐름이 발생했다”를 증명하는 모델은 아니다. 사건 날짜를 기준으로 자금 흐름이 어떻게 달라졌는지를 비교하는 **이벤트 스터디 방식의 탐색 분석**에 가깝다.

### 19.2 고래 송금 데이터는 주간 단위

고래 송금 데이터는 주간 단위라서 일 단위 이벤트 윈도우에 붙일 때 7일로 균등 분배했다. 따라서 특정 하루의 실제 고래 송금액을 완전히 복원한 값이라기보다, 주간 흐름을 일 단위 분석에 맞춘 근사값이다.

### 19.3 거래소 입출금은 해석이 필요함

거래소 입금은 매도 압력 신호로 볼 수 있지만, 항상 실제 매도를 의미하지는 않는다. 거래소 출금도 장기 보유 신호일 수 있지만, 단순 지갑 이동일 수도 있다. 그래서 본 프로젝트는 입금/출금/가격/뉴스를 함께 해석한다.

### 19.4 BTC 가격은 현재 보조 데이터

BTC 가격은 파이프라인과 DB, API에는 포함되어 있지만, 현재 대시보드의 핵심 해석은 ETH 가격 중심이다. BTC 가격은 향후 “ETH만의 반응인지, 시장 전체 반응인지”를 비교하기 위한 확장 데이터로 남아 있다.

---

## 20. 핵심 요약

이 프로젝트의 데이터 분석은 다음 네 단계로 요약할 수 있다.

1. **원천 데이터 정제**  
   고래 송금, 거래소 입출금, 가격, 뉴스, 심리 데이터를 날짜 기준으로 정리한다.

2. **이벤트 윈도우 생성**  
   각 사건의 전후 7일 데이터를 하나의 분석 단위로 묶는다.

3. **기준선 대비 변화율 계산**  
   사건 전 7일 평균을 기준으로 사건일과 사건 후 반응을 비교한다.

4. **가설 검증과 시각화**  
   입금 증가, 출금 우위, 자금 재배치, 가격 반응 같은 가설을 규칙 기반으로 판정하고 대시보드에서 보여준다.

따라서 이 프로젝트는 단순히 차트를 그린 것이 아니라, **시장 이벤트 → 자금 흐름 변화 → 가설 검증 → 시각화**로 이어지는 데이터 분석 파이프라인을 구현한 프로젝트다.
