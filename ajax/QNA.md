# WeatherEats 예상 기술 질문 + 모범 답변

## AJAX / jQuery

### Q1. `$.ajax()`와 `fetch()`의 차이점은 무엇이고, 왜 `$.ajax()`를 선택했나요?

`fetch()`는 브라우저 내장 API로 Promise 기반이고, `$.ajax()`는 jQuery 라이브러리가 제공하는 메서드로 jQuery Deferred 기반입니다. 이번 프로젝트에서는 DOM 조작에 이미 jQuery를 사용하고 있었기 때문에, 별도의 라이브러리 없이 `$.ajax()`로 통일하는 것이 일관성 있다고 판단했습니다. 또한 `$.when()`을 활용한 병렬 처리, `.done()/.fail()` 체이닝 등 jQuery Deferred 패턴이 여러 API를 조합하는 데 편리했습니다. `fetch()`는 네트워크 에러만 reject하고 HTTP 4xx/5xx는 reject하지 않아 별도 처리가 필요한 반면, `$.ajax()`는 HTTP 에러 시 자동으로 `.fail()`로 빠지는 점도 장점이었습니다.

---

### Q2. `$.when()`으로 병렬 처리할 때, 하나의 요청이 실패하면 나머지 요청은 어떻게 되나요?

`$.when()`은 하나라도 reject되면 즉시 `.fail()` 콜백을 실행합니다. 나머지 요청은 취소되지 않고 백그라운드에서 계속 진행되지만, 그 결과는 무시됩니다. 이 프로젝트에서는 역지오코딩 요청을 내부에서 try-catch처럼 래핑해서, 실패해도 `null`로 resolve하도록 처리했습니다. 그래서 Kakao API가 장애가 나도 날씨 정보는 정상적으로 렌더링됩니다. 핵심 요청(날씨, 예보)이 실패한 경우에만 `.fail()`에서 에러 배너를 표시합니다.

---

### Q3. AJAX 요청 시 CORS 문제를 겪은 적 있나요? 어떻게 해결했나요?

OpenWeatherMap과 Kakao API 모두 서버 측에서 CORS 헤더를 허용하고 있어서 직접적인 CORS 문제는 없었습니다. Kakao Local API의 경우 REST API 키를 `Authorization` 헤더에 담아 보내는데, 이때 브라우저가 preflight 요청(OPTIONS)을 먼저 보내게 됩니다. Kakao 서버가 이를 허용하고 있어서 정상 동작합니다. 만약 CORS가 막혔다면 서버 프록시를 두거나 JSONP 방식을 고려했을 것입니다.

---

### Q4. `$.ajax()`의 `dataType` 옵션을 지정하지 않으면 어떻게 동작하나요?

`dataType`을 생략하면 jQuery가 응답의 `Content-Type` 헤더를 보고 자동으로 추론합니다. `application/json`이면 JSON으로 파싱하고, `text/html`이면 문자열로 반환합니다. 이 프로젝트에서는 OpenWeatherMap과 Kakao API 모두 JSON을 반환하므로 `dataType: 'json'`을 명시하지 않아도 동작하지만, 명시적으로 지정하면 응답이 JSON이 아닐 때 `.fail()`로 빠지게 되어 더 안전합니다.

## API / 데이터

### Q5. OpenWeatherMap API 키가 클라이언트에 노출되는데, 보안 문제는 없나요? 어떻게 관리했나요?

맞습니다. 클라이언트 사이드 앱이라 API 키가 브라우저에 노출됩니다. 이를 최소화하기 위해 소스 코드에는 `config.js`에 플레이스홀더만 두고, GitHub Actions 배포 시 `secrets`에 저장된 실제 키를 주입하는 방식을 사용했습니다. 워크플로우에서 `cat > _site/js/config.js`로 빌드 시점에 키를 삽입합니다. 완전한 보안은 아니지만, 소스 코드 저장소에 키가 커밋되는 것은 방지됩니다. 더 안전하게 하려면 백엔드 프록시 서버를 두어 키를 서버에서만 관리하는 방법이 있습니다.

---

### Q6. Kakao API 호출 시 일일 요청 제한이 있는데, 이에 대한 대응은 했나요?

Kakao API가 제한에 걸리거나 403 에러를 반환하면, 코드에서 이를 감지하여 **데모 모드로 전환**합니다. `kakao.js`에서 검색 실패 시 12개의 데모 음식점 데이터를 자동 생성하여, API가 안 되더라도 기본적인 UI와 기능을 시연할 수 있도록 했습니다. 사용자에게는 "데모 데이터입니다"라는 안내를 표시합니다.

---

### Q7. 역지오코딩에서 행정동(H)을 법정동(B)보다 우선한 이유는 무엇인가요?

Kakao `coord2regioncode` API 응답에는 행정동(H 타입)과 법정동(B 타입) 두 종류가 포함됩니다. 행정동은 실제 행정 업무 기준 구역으로 사람들이 일상적으로 사용하는 지역명에 더 가깝습니다. 예를 들어 "평촌동"이라고 하면 대부분 행정동 기준입니다. 법정동은 법적 주소 기준이라 사용자에게 생소할 수 있어서, 사용자 친화적인 행정동을 우선 선택했습니다.

---

### Q8. 날씨 데이터와 메뉴 추천 로직은 어떤 기준으로 매핑했나요?

`weather.js`에서 날씨 조건과 기온을 조합하여 추천합니다. 먼저 날씨 상태를 확인해서, 비(Rain/Drizzle)면 파전/막걸리/해물탕, 눈(Snow)이면 국밥/찌개/샤브샤브를 추천합니다. 그 외에는 기온 구간별로 나눕니다. 5도 이하면 따뜻한 국물 요리, 12도 이하면 라면/칼국수, 22도 이하면 한식/김밥, 30도 이하면 냉면/초밥/샐러드, 30도 초과면 빙수/아이스크림입니다. 추가로 습도 80% 이상이면 습한 날씨 맥락을, 풍속 10m/s 이상이면 바람 맥락을 더합니다.

## Geolocation / 위치

### Q9. Geolocation API를 사용자가 거부하면 어떻게 처리하나요?

위치 권한이 거부되면 `.fail()` 콜백에서 **기본 좌표로 폴백**합니다. `common.js`에 정의된 기본 좌표(경기도 안양시, 위도 37.3943, 경도 126.9568)를 사용하여 해당 지역 기준으로 날씨와 맛집을 표시합니다. 동시에 "위치 정보를 가져올 수 없어 기본 위치를 사용합니다"라는 안내 배너를 보여주고, 수동으로 위도/경도를 입력할 수 있는 폼도 제공합니다.

---

### Q10. 2단계 위치 획득에서 정확도 기준을 1,200m로 잡은 이유가 있나요?

실내 환경에서 GPS 없이 Wi-Fi나 셀룰러 기반으로 위치를 잡으면 보통 1~2km 오차가 발생합니다. 테스트 결과 1,200m 이하면 동네 수준의 맛집 검색에 충분히 유의미했고, 이 이상이면 다른 동네 결과가 섞일 수 있어서 이 기준을 잡았습니다. 2차 시도에서는 타임아웃을 12초에서 20초로 늘려 GPS 위성 신호를 확보할 시간을 더 줍니다. 그래도 1,000m 이상이면 `shouldPromptManualLocation()`이 true를 반환하여 수동 입력을 유도합니다.

---

### Q11. HTTPS 환경이 아니면 Geolocation API가 동작하지 않는데, 개발 시 어떻게 테스트했나요?

크롬은 `localhost`에서는 HTTP여도 Geolocation을 허용합니다. 로컬 개발 시에는 `localhost`로 접속하여 테스트했고, 배포 환경은 GitHub Pages가 기본적으로 HTTPS를 제공하므로 문제가 없었습니다. 만약 다른 기기에서 테스트할 때는 크롬 DevTools의 Sensors 탭에서 위치를 임의로 설정하거나, 수동 좌표 입력 폼을 활용했습니다.

---

## localStorage / 상태 관리

### Q12. localStorage 대신 sessionStorage나 IndexedDB를 고려하지 않은 이유는?

sessionStorage는 브라우저 탭을 닫으면 데이터가 사라집니다. 즐겨찾기는 다음에 다시 접속해도 유지되어야 하므로 적합하지 않습니다. IndexedDB는 대용량 데이터에 적합하지만, 즐겨찾기 최대 50개, 최근 검색어 6개 수준의 소량 데이터를 저장하는 데는 오버엔지니어링입니다. localStorage의 약 5MB 용량이면 충분하고, API도 `setItem/getItem`으로 단순해서 이 프로젝트 규모에 가장 적합했습니다.

---

### Q13. localStorage에 저장할 수 있는 용량 제한은 얼마이고, `QuotaExceededError` 외에 다른 예외 상황은 고려했나요?

브라우저별로 다르지만 대부분 약 5MB입니다. `QuotaExceededError` 외에도, 시크릿 모드에서 localStorage 자체가 비활성화되거나 `setItem`이 에러를 던지는 경우가 있습니다. `writeFavorites()` 함수에서 try-catch로 감싸서, 저장 실패 시 가장 오래된 항목을 제거하며 재시도하고, 빈 배열에서도 실패하면 사용자에게 저장 불가 안내를 합니다. 시크릿 모드에서는 저장이 안 될 수 있지만, 조회는 가능하도록 읽기 로직에서도 별도로 try-catch 처리를 했습니다.

---

### Q14. 즐겨찾기 데이터 구조는 어떻게 설계했나요? 어떤 필드를 저장하나요?

즐겨찾기는 JSON 배열로 저장합니다. 각 항목에는 장소 ID, 장소명, 카테고리, 주소, 거리, 전화번호, 카카오맵 URL, 그리고 저장 시각(timestamp)을 포함합니다. 저장 시각을 넣은 이유는 즐겨찾기 목록을 최신순으로 정렬하기 위해서입니다. 최대 50개까지 저장하도록 제한을 두었고, 이를 초과하면 가장 오래된 항목이 자동으로 제거됩니다.

---

## 지도 / Kakao Maps

### Q15. Kakao Maps SDK를 동적 로딩한 이유는 무엇이고, 정적 로딩 대비 어떤 장단점이 있나요?

메인 페이지와 즐겨찾기 페이지에서는 지도가 필요 없습니다. SDK를 모든 페이지에 정적으로 넣으면 불필요한 리소스를 로드하게 됩니다. 동적 로딩하면 검색 페이지에 진입할 때만 SDK를 로드하므로 초기 페이지 로딩 속도가 빨라집니다. 단점은 SDK 로딩 시간만큼 지도 표시가 지연될 수 있다는 점인데, `autoload=false`와 Deferred 패턴으로 로딩 완료 시점을 정확히 제어하고, 한 번 로드한 SDK는 `sdkPromise`에 캐싱하여 중복 로딩을 방지합니다.

---

### Q16. 지도 마커가 많아지면 성능 문제가 생길 수 있는데, 클러스터링 같은 최적화를 고려했나요?

현재는 페이지당 최대 10개의 검색 결과만 표시하고, 전체적으로도 최대 45개(Kakao API 제한)이기 때문에 마커 수가 많지 않아 클러스터링은 적용하지 않았습니다. 만약 마커가 수백 개 이상이 된다면 Kakao Maps의 MarkerClusterer 라이브러리를 추가하거나, 지도 줌 레벨에 따라 마커를 동적으로 표시/숨김 처리하는 방법을 고려할 것입니다.

---

### Q17. 지도-리스트 양방향 연동에서 `highlightCard()` 하나로 통합한 이유는? 관심사 분리 측면에서 문제는 없나요?

마커 클릭과 리스트 호버 모두 같은 동작(카드 하이라이트 + 마커 강조 + 인포윈도우 표시)을 수행해야 합니다. 이걸 분리하면 동일한 로직이 두 곳에 중복되어 유지보수가 어려워집니다. `highlightCard()`에 `shouldScroll` 파라미터를 두어, 마커 클릭 시에는 리스트 스크롤을 하고, 리스트 호버 시에는 스크롤하지 않도록 분기했습니다. 하나의 함수에서 처리하되 호출자에 따라 동작을 미세 조정하는 방식입니다.

## 아키텍처 / 설계

### Q18. 모듈을 `common.js`, `weather.js`, `kakao.js` 등으로 분리한 기준은 무엇인가요? ES6 모듈(`import/export`) 대신 전역 객체를 사용한 이유는?

기능 도메인별로 분리했습니다. 공통 유틸은 `common.js`, 날씨 관련은 `weather.js`, 카카오 지도/검색은 `kakao.js`, 즐겨찾기 CRUD는 `favorites.js`, 그리고 페이지별 컨트롤러로 `main.js`, `search.js`를 두었습니다. ES6 모듈을 사용하지 않은 이유는, 이번 과제가 jQuery + AJAX 중심의 전통적인 프론트엔드 구현이 목적이었기 때문입니다. 번들러(Webpack 등) 없이 `<script>` 태그로 로드하는 구조에서는 `window.WeatherEats`라는 네임스페이스 객체에 각 모듈을 등록하는 IIFE 패턴이 깔끔합니다.

---

### Q19. SPA가 아닌 MPA(멀티 페이지) 구조를 선택한 이유는 무엇인가요?

이 프로젝트의 핵심은 jQuery와 AJAX 활용입니다. SPA를 구현하려면 라우팅, 상태 관리, 히스토리 API 등 추가 복잡도가 생기고, React나 Vue 같은 프레임워크가 필요해집니다. MPA 구조에서는 각 페이지가 독립적이어서 구현이 단순하고, 페이지 간 데이터 전달은 URL 파라미터로 처리하면 충분합니다. 또한 GitHub Pages에 배포할 때 별도의 서버 사이드 라우팅 설정이 필요 없다는 장점도 있습니다. 404.html을 두어 SPA 라우팅 폴백도 고려했지만, 기본적으로는 MPA가 이 과제의 요구사항에 가장 적합했습니다.

---

### Q20. GitHub Actions로 배포를 자동화했다고 했는데, CI/CD 파이프라인 구성은 어떻게 되어 있나요? API 키는 어떻게 주입하나요?

`main` 브랜치에 push하면 자동으로 워크플로우가 실행됩니다. 순서는 이렇습니다. 먼저 소스를 checkout하고, `rsync`로 `.git`, `.github`, `node_modules` 등을 제외한 파일을 `_site/` 디렉토리에 복사합니다. 그 다음 GitHub Secrets에 저장된 `WEATHER_API_KEY`, `KAKAO_REST_KEY`, `KAKAO_JS_KEY`를 환경변수로 읽어와서, `cat` 명령으로 `_site/js/config.js` 파일을 동적으로 생성합니다. 이렇게 하면 소스 코드에는 키가 없고, 빌드 시점에만 주입됩니다. 마지막으로 `upload-pages-artifact`와 `deploy-pages` 액션으로 GitHub Pages에 배포합니다.
